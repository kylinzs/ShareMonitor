## Context

- 现象：在个股详情页切换到“东方财富”数据源后，最新价/涨跌/开高低/成交量长期显示占位符（`--`），且没有明显错误提示；切换到“腾讯”数据源则可正常显示。
- 当前数据链路：`DetailScreen` 通过 `DetailViewModel` 订阅 `QuotesRepository` 的 `quoteCache` 与历史缓存；刷新时会调用 `refreshQuote` 与 `refreshHistory`。
- 关键问题：东方财富接口可能返回“data 存在但关键字段为空”的结构；如果将其当作有效 Quote 写入缓存，会导致 UI 认为“有 quote”但字段全空，从而既不触发错误提示，也不会触发回退展示。

## Goals / Non-Goals

**Goals:**

- 当报价缺少关键字段（例如 `lastPrice == null`）时，不把该报价当作有效数据写入缓存，避免详情页静默空白。
- 报价刷新若得到空结果（例如 `getQuotes` 返回空列表），将其视为失败并提供可提示的信息。
- 在详情页，当报价不可用但分时历史可用时，推导并展示最小可用的价格信息（`last/open/high/low`），并显示提示信息。

**Non-Goals:**

- 不引入更复杂的“交易日历/是否开盘”判断（当前仅做弱判断与 UI 兜底）。
- 不重构数据源协议或引入新的远端接口；只做数据质量校验与 UI 兜底策略优化。
- 不对“成交量单位/字段定义差异”做统一换算（仍以数据源返回为准，仅保证不静默空白）。

## Decisions

1) **数据源层：关键字段为空即视为无报价**

- 选择：在 Eastmoney 数据源映射处，如果 `lastPrice` 为空，直接返回 `null`（表示“无有效报价”）。
- 理由：避免产生“Quote 对象存在但核心字段为空”的半有效状态，导致 UI 无法分辨并且无法触发回退或错误提示。
- 备选：返回 Quote 但增加额外标记字段（例如 `isValid`）。缺点是需要修改 domain model，影响面更大；不采纳。

2) **仓储层：空结果视为失败**

- 选择：`refreshQuote(s)` 在数据源成功返回但结果为空时，视为失败并返回可提示的错误信息。
- 理由：空列表在业务语义上等同于“未拿到数据”，如果仍返回 Success，会导致 UI 与用户误判。
- 备选：继续返回 Success，但在 UI 端检测空缓存并提示。缺点是分散逻辑且不同页面行为不一致；不采纳。

3) **UI 层：以 `lastPrice` 作为可用性判断，并提供分时推导兜底**

- 选择：详情页仅把 `lastPrice != null` 的 Quote 当作“可用报价”；当报价不可用且分时序列存在时，用分时 close 序列推导 `open/high/low/last` 并提示“报价不可用，已从分时推导”。
- 理由：回退策略集中在 ViewModel，能确保 UI 不会因为数据源异常而长期空白；并且推导策略可测试、可解释。
- 备选：在 Compose UI 层做推导。缺点是 UI 复杂度上升且难以复用；不采纳。

## Risks / Trade-offs

- [风险] Eastmoney 的字段含义/返回结构变化导致“lastPrice 为空”误判 → 缓解：允许通过历史序列兜底展示，同时刷新失败会有提示，便于排障。
- [风险] 分时兜底不包含 `change/changePct/volume` → 缓解：提示“推导”来源，并在可用时优先使用真实报价。
- [取舍] 不引入交易日历，可能在极端情况下误判“当日不可用” → 影响：仅影响提示/回退路径；整体体验仍优于静默空白。
