## Context

- 项目：ShareMonitor（Android/Kotlin/Compose）。
- 现状：代码整体缺少注释与文档化说明，阅读成本高；并且缺少“新增文件需要注释”的规范约束，后续很容易回到“无注释”状态。
- 约束：
  - 不引入大规模重构、不改变业务行为，主要做“可读性增强”。
  - 注释以中文为主，便于团队沟通；同时避免对显而易见的代码进行噪音式注释。

## Goals / Non-Goals

**Goals:**

- 为核心代码补齐中文注释，形成“读代码即可理解意图/边界”的最低保障。
- 建立可执行的项目代码规范文档，并新增规则：**以后新增的代码文件均要加上中文注释**（至少文件级 KDoc + 对外 API 的 KDoc）。
- 通过编译与基础测试验证，确保注释改动不引入语法/导入/格式问题。

**Non-Goals:**

- 不在本变更内引入/配置强制化的静态检查工具链（如 detekt 自定义规则、CI gate）。如果后续需要强制执行，可单独立项。
- 不追求“每一行都有注释”，不把注释当作重复代码的翻译；不为临时/显而易见的 UI 排版细节写冗余注释。

## Decisions

1) **注释类型以 KDoc 为主**

- 对象：对外可复用/跨层理解成本高的结构（domain model、Repository、DataSource、ViewModel、DI 容器、关键入口 Composable）。
- 做法：
  - class/interface/object/top-level function：补充 `/** ... */` KDoc（中文）。
  - 对关键字段/参数使用 `@property`/`@param` 简短解释。
  - 仅在“实现意图不明显”处补少量行内注释（例如：缓存/节流/回退逻辑、数据源兼容处理）。
- 备选方案：大量行内注释或块注释。缺点是噪音高、易与实现偏离；因此不采纳。

2) **新增文件必须有文件级中文注释**

- 选择在“代码规范文档”中固化此规则（单一事实来源），并在开发流程/评审 checklist 中执行。
- 最小要求建议：
  - 文件顶部（或主要 public 类型上）提供“本文件用途”中文 KDoc。
  - public API（跨包/跨层调用）必须提供中文 KDoc。
- 备选方案：只要求 public API 注释。缺点是新文件整体意图仍缺失；因此不采纳。

3) **规范文档落地位置**

- 新增 `CODE_STYLE.md`（或 `docs/code-style.md`）作为项目代码规范入口，并在需要处（如 README/团队 wiki/IDE 配置说明）引用它。
- 选择靠近仓库根目录，便于发现与维护。

## Risks / Trade-offs

- [风险] 注释覆盖面广导致合并冲突或 review 负担增加 → [缓解] 分批提交/按模块分阶段；每批都保证可编译、测试通过。
- [风险] 注释与实现长期偏离 → [缓解] 注释聚焦“不变的意图/边界”，避免记录易变的细节；把示例/约定放在规范文档集中维护。
- [取舍] 不做强制工具校验 → [影响] 依赖评审执行力 → [缓解] 后续可独立引入 detekt/ktlint 规则或 pre-commit/CI 检查。
